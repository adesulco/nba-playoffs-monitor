## Stream A — framing consistency

Before v0.11.8, the top content block on each page used a different padding:

| Page | Top block padding |
|---|---|
| Home grid | `16px 20px 20px` |
| EPL / F1 / Tennis hero | `20px 20px 14px` |
| ComingSoon (FIFA / Liga 1) hero | `36px 24px 32px` |
| Recap date-masthead | `22px 24px` |

The effect: flicking between Home and a sport caused an 8–20 px vertical jolt and a 4 px horizontal shift. v0.11.2's "same masthead everywhere" win was undercut by this — the chrome was consistent, the content frame wasn't.

### Fix: three shared utility classes in `src/index.css`

```css
.dashboard-hero          { padding: 24px 24px 20px; }       /* desktop */
.dashboard-hero--bordered { border-bottom: 1px solid var(--line); }
.dashboard-section       { padding: 20px 24px; }             /* subsequent blocks */

@media (max-width: 720px) {
  .dashboard-hero    { padding: 20px 16px 16px; }
  .dashboard-section { padding: 16px; }
}
```

### Pages wired to the class

- **Home.jsx** — `.home-dashboard-grid` now carries `.dashboard-hero` (padding via class, inline `padding` removed).
- **EPL.jsx / F1.jsx / Tennis.jsx** — sport hero block swapped from inline `20px 20px 14px` to `.dashboard-hero`. Gradient background stays inline (per-sport accent).
- **ComingSoon.jsx** — hero uses `.dashboard-hero .dashboard-hero--bordered` (was inline `36px 24px 32px` + inline `borderBottom`).
- **Recap.jsx** — date-masthead uses `.dashboard-hero .dashboard-hero--bordered` (was inline `22px 24px` + inline `borderBottom`).

Result: the first content row sits at exactly `24px` from the sub-row on every desktop page, `20px` on mobile. Switching sports now feels like a body swap inside the same frame, not a page reload.

## Stream B — Sprint 3 (saved teams + URL state)

### 1. `useQueryParamSync` — two-way bind between URL and state

New hook at `src/hooks/useQueryParamSync.js` (~50 LOC). Reads a query param on first mount → calls a setter if value differs. Writes to URL whenever state changes via `history.replaceState` (no nav, no scroll jump). Removes the param when state becomes null.

```js
useQueryParamSync('club', selectedEPLClub, setSelectedEPLClub);
```

Wired to three sport pickers:

- **EPL** — `?club=arsenal`
- **F1** — `?team=mclaren`
- **Tennis** — `?player=alcaraz`

Share a link with the param → recipient's dashboard opens with the same picker pre-applied. Per the audit's Persona B Diandra "filter state not in URL" gap (§04 moment 05). NBA fav isn't URL-encoded this ship; the NBA dashboard has per-team pages (`/nba-playoff-2026/lakers`) for that purpose.

### 2. `MilikmuStrip` on Home

New component at `src/components/MilikmuStrip.jsx`. Reads:

- NBA fav from `localStorage['gibol:favTeam']` (subscribes to `storage` + `focus` events for same-tab updates).
- EPL / F1 / Tennis picks from `useApp()` (already in context).

Renders a horizontal row of up to 4 chips above the dashboard grid:

- Each chip: 22 px accent square with 3-letter abbr, team short name, sport tag in mono caps.
- Each chip links to its sport dashboard with the picker pre-filtered via the URL params from stream B #1.
- Section header: "★ MILIKMU · {N} TIM" (ID) / "★ YOURS · {N} TEAMS" (EN).
- **Renders null** when the user has zero pins — first-time visitors see the gateway grid unchanged, zero clutter. This keeps v0.11.8 backward-compatible by construction.

### 3. What's NOT in this ship

- **Share-as-URL button on desktop.** The URL state exists; adding a visible "Copy link" affordance is a 30-minute polish I'd rather bundle with the broader editorial-share-card flow in Sprint 5.
- **EPL/F1/Tennis hooks wired to SWR cache.** Pattern proven in v0.11.7 on `usePlayoffData`; porting = ~1 day of mechanical work. Slotted for Sprint 4 opening.
