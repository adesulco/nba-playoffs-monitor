## What shipped

### 1. `src/lib/swrCache.js` — generic localStorage SWR helpers

~150 LOC. Stateless, no React deps. Three functions:

```js
readCache('nba-scoreboard', { ttlMs: 120_000 }) // → null if missing/stale/parse-fail
writeCache('nba-scoreboard', data)               // silent on quota / private-mode
clearCache('nba-scoreboard')                     // explicit invalidation
readCacheMeta('nba-scoreboard')                  // { t, ageMs } for "last updated 2m ago" chrome
```

Design notes:

- **Namespaced keys** under `gibol:swr:v1:<name>`. The `v1` segment is a schema version — bump to auto-invalidate everything if the cached shape ever changes incompatibly.
- **TTL per read**, not per write. Lets one cached payload serve many callers with different tolerances for staleness.
- **Quota-safe** — every write is try/catch'd; iOS private-mode `QuotaExceededError` is swallowed silently (falls back to memory-only behaviour).
- **Size-capped** — entries over ~200 KB are rejected so one bad payload can't eat the 5–10 MB per-origin budget.

### 2. `usePlayoffData` wired to SWR

The NBA hook now hydrates its four stateful payloads from cache synchronously in the `useState` initialisers:

| State | Cache key | TTL | Rationale |
|---|---|---|---|
| `champion` | `nba-champion` | 15 min | Odds drift slowly outside trade windows |
| `mvp` | `nba-mvp` | 24 h | MVP picks move ~weekly during playoffs |
| `games` | `nba-scoreboard` | 2 min | Live scores — tight ceiling |
| `gamesByDay` | `nba-scoreboard-range` | 5 min | 7-day window, refreshes on tab wake |

Each `refresh()` call that succeeds writes back to cache. Initial status is now `'stale'` (not `'connecting'`) when cache was hydrated, so the status dot reads correctly. First-ever visits keep `'connecting'` behaviour.

**Rangga's moment 01** in the audit ("blank cream screen for ~3 seconds") now paints the last-known scoreboard + champion + MVP data on the first frame, then quietly refreshes in the background. Cold-cache users get the old behaviour — they'll hit this once, then become return-visit users.

### 3. `public/sw.js` — Polymarket proxy cached

Added to the `fetch` handler:

```js
if (url.pathname.startsWith("/api/proxy/")) {
  event.respondWith(networkFirst(request, RUNTIME_CACHE, 5000));
  return;
}
```

Polymarket calls now survive a patchy connection — if the 5-second network race fails, the SW serves the last-cached proxy response instead of erroring out.

ESPN calls remain out of scope here: they're direct cross-origin to `site.api.espn.com`, and SW can't meaningfully serve cross-origin opaque responses. The localStorage SWR layer in change #2 covers that path.

`SW_VERSION` bumped from `gibol-2026-04-18-v1` to `gibol-2026-04-23-v2` to force all existing clients to flush old caches on their next visit.

## Verify after deploy

1. **Cold visit.** Clear localStorage (`Application → Storage → Clear`), hard-refresh `gibol.co/nba-playoff-2026`. FCP behaviour is identical to v0.11.6 — scoreboard paints after ESPN responds.
2. **Return visit.** Close the tab, reopen. DevTools → Network → slow-3G profile. The scoreboard and champion odds should paint **immediately** — before ESPN responds. Background refresh then overwrites within ~2–3 seconds.
3. **Verify cache keys.** DevTools → Application → Local Storage → `https://www.gibol.co`. Should see four keys under `gibol:swr:v1:` — `nba-champion`, `nba-mvp`, `nba-scoreboard`, `nba-scoreboard-range`. Each value is `{"t": <ms-timestamp>, "v": <payload>}`.
4. **Offline proxy resilience.** DevTools → Network → Offline. Reload. Champion odds should still render (SW serves from `gibol-2026-04-23-v2-runtime` cache); ESPN-backed scoreboard falls back to localStorage. Confirm no "undefined" / crash screens.
5. **SW version bump.** DevTools → Application → Service Workers. New version `gibol-2026-04-23-v2` should replace `-v1` and purge old caches on activate.

## Known gaps (deliberately deferred)

- **EPL / F1 / Tennis hooks** still fire without SWR cache. Pattern is proven in `usePlayoffData`; porting to the other sport hooks is ~1 day of mechanical work. Slotted for Sprint 2.5 or Sprint 3 opening.
- **ESPN direct calls** don't benefit from SW caching (CORS-opaque). Routing them through `/api/proxy/*` is a broader refactor (~2 days) — worthwhile once we add a second sport that also hits ESPN directly (soccer, tennis).
- **Background sync / periodic revalidation.** SW could refresh cached scores every N minutes even when the tab is closed. Deferred until we have actual DAU patterns to tune against.
