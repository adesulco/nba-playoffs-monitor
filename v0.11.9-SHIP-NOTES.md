# v0.11.9 — Sprint 4 week-1: thumb-first mobile nav

**Date:** 2026-04-23
**Branch:** main
**Risk:** LOW. Pure additive UX: one new component, one small CSS block, one CustomEvent listener on V2TopBar. Nothing on the desktop surface changes.
**Audit ref:** §07 Sprint 4 ("Bottom nav introduced: Skor · Jadwal · Saya"), §04 Persona A Rangga moment 05 + Persona B Diandra moment 02 (command-palette discovery gap on mobile).

---

## Why

The audit's mobile scorecard (§03, 48/100, F) called out two things this ship addresses:

1. **"No bottom-nav or FAB — all navigation is top-weighted."** On a 375 px viewport, the V2TopBar nav scrolls horizontally but users rarely discover the swipe. Mobile users need at least one always-thumb-reachable destination.
2. **"Command palette (⌘K) is hidden."** Crown-jewel feature that mobile users can't invoke (no keyboard). Zero mobile discovery. Audit called it out in two separate persona walkthroughs.

One bottom-nav solves both.

---

## What shipped

### `<MobileBottomNav />` — new component

`src/components/MobileBottomNav.jsx`, ~150 LOC. Three persistent thumb-sized buttons:

| # | Icon | Label (ID/EN) | Action |
|---|---|---|---|
| 1 | 🏠 Home | Home / Home | `<Link to="/">` |
| 2 | 🔍 Magnifier | Cari / Search | Dispatches `gibol:open-search` |
| 3 | 🏆 Bracket | Bracket / Bracket | `<Link to="/bracket">` |

Each button is 52 px tall min, flex-1 wide (~33% of viewport), uppercase mono label below SVG icon. Active page gets a 24×2 px amber tick at the top edge of the button + `aria-current="page"`. Inactive color: `var(--ink-3)`; active: `var(--ink)`. `WebkitTapHighlightColor: transparent` so iOS doesn't flash grey on tap.

### Mobile-only rendering via CSS

`src/index.css`:

```css
.mobile-bottom-nav { display: flex; }
@media (min-width: 721px) {
  .mobile-bottom-nav { display: none !important; }
}
@media (max-width: 720px) {
  body { padding-bottom: calc(60px + env(safe-area-inset-bottom, 0)); }
}
```

The body gets a reserved gutter on mobile so content never slips behind the fixed nav. iOS `safe-area-inset-bottom` is added on top of the 60 px so the home-indicator area doesn't crop the last button.

Mounted once in `src/App.jsx` after `<main>` so it sits above every route without per-page wiring.

### `gibol:open-search` CustomEvent handshake

V2TopBar already managed the search palette via `searchOpen` state. BottomNav can't reach that state directly without coupling. Fix: V2TopBar adds a `window` event listener:

```js
function onOpenSearch() { setSearchOpen(true); }
window.addEventListener('gibol:open-search', onOpenSearch);
```

BottomNav fires:

```js
window.dispatchEvent(new CustomEvent('gibol:open-search'));
```

No shared state, no context, no prop drilling. Future mobile FAB or deep-link (`?search=1`) can trigger the same event.

## Verify after deploy

1. **Desktop (≥721 px).** The bottom nav should NOT render. Body should have no reserved bottom padding. V2TopBar behaviour unchanged.
2. **Mobile (Chrome DevTools → iPhone SE, 375 px).** Fixed bottom bar appears with three icons. Home is highlighted (amber tick) on `/`, Bracket highlighted on `/bracket`. Content scrolls behind nothing — the body's 60 px reserve holds.
3. **Search from mobile.** Tap the magnifier. Search palette opens (same palette ⌘K triggers on desktop). ⎋ or tap outside closes.
4. **iOS home-indicator.** On an actual iPhone 14+ or Safari responsive-design mode with iPhone 15, the nav sits above the home indicator with a subtle extra gutter (`env(safe-area-inset-bottom)`).
5. **Accessibility.** VoiceOver rotor → "Landmarks" now lists the navigation as well. Buttons announce "Search, button" / "Home, link" with `aria-current="page"` when active.
